# Аудит проекта Nim2Book Mobile

Как Senior Flutter Developer, я провел анализ архитектуры и кода вашего проекта. В целом, проект использует современный стек технологий (Riverpod, GoRouter, Freezed, Dio) и имеет хорошие настройки линтера. Однако есть несколько архитектурных проблем и моментов, которые стоит улучшить для масштабируемости и поддержки.

## 1. Архитектура и Структура папок

**Проблема:**
В проекте наблюдается смешивание подходов "Layer-first" (папки по типу: screens, providers) и "Feature-first" (папка features).
Например, существует папка `lib/screens` и отдельно `lib/features`. Это создает путаницу:
- `lib/features/learning_session_screen` содержит логику и виджеты.
- `lib/screens/learning_session_screen` содержит сам экран.

**Рекомендация:**
Полностью перейти на **Feature-first** архитектуру. Папка `screens` должна исчезнуть, а экраны должны переместиться внутрь соответствующих фич.
Пример правильной структуры:
```
lib/
  features/
    auth/
      data/ (api, models, repositories)
      domain/ (entities, usecases - опционально)
      presentation/ (providers, screens, widgets)
    learning/
      presentation/
        screens/
          learning_session_screen.dart
        widgets/
          ...
        providers/
          ...
```

## 2. Обработка ошибок (Critical)

**Проблема:**
В `AuthNotifier` (и вероятно в других местах) ошибки "проглатываются".
```dart
catch (_) {
  state = const AuthState.unauthenticated();
  return false;
}
```
Это плохая практика. UI не узнает, почему произошла ошибка (нет интернета, неверный пароль, ошибка сервера). Пользователь просто увидит, что ничего не произошло или его выкинуло.

**Рекомендация:**
1. Использовать `AsyncValue` от Riverpod для управления состоянием загрузки и ошибок.
2. Или возвращать специальные типы-обертки, например `Result<T, Exception>` (можно использовать пакет `fpdart` или `dartz`), чтобы прокидывать конкретную ошибку в UI для отображения SnackBar или Dialog.

## 3. Роутинг

**Проблема:**
В `router.dart` используются хардкод-строки для путей (`/login`, `/my-books`). При рефакторинге или опечатках это приведет к багам.

**Рекомендация:**
Вынести все пути в отдельный класс констант или Enum.
```dart
abstract class AppRoutes {
  static const login = '/login';
  static const myBooks = '/my-books';
  // ...
}
```
Еще лучше — использовать кодогенерацию для роутов (например, `go_router_builder`), чтобы сделать навигацию типобезопасной.

## 4. Инициализация приложения

**Проблема:**
В `main.dart` используется `Future.microtask` для инициализации провайдеров:
```dart
Future.microtask(() async {
  await ref.read(authNotifierProvider.notifier).initialize();
  // ...
});
```
Это "костыль", который может привести к гонкам состояний (race conditions), когда UI начинает строиться до того, как данные готовы.

**Рекомендация:**
Использовать `AppStartupWidget` паттерн. Создать виджет, который оборачивает все приложение и ждет инициализации всех критических сервисов, показывая сплэш-скрин, пока `Future.wait([...])` не выполнится.

## 5. API Клиент

**Проблема:**
Логика обновления токена (`refreshToken`) реализована вручную через `Completer` и интерцепторы. Это сложное место, где легко допустить ошибку с конкурентностью запросов.

**Рекомендация:**
Рассмотреть использование готовых решений для Dio, например `dio_smart_retry` или проверить реализацию через `Fresh` (пакет `fresh_dio`), который специально создан для обработки OAuth2 токенов и их обновления.

## 6. Зависимости и Utils

- **Assets:** Не видно использования кодогенерации для ассетов (`flutter_gen`). Рекомендуется использовать, чтобы вместо строк `assets/icon/icon.png` писать `Assets.icon.icon`.
- **Localization:** Используется стандартный `l10n`, что хорошо.

## Итог

Проект написан на высоком уровне, код чистый, используется строгий линтер. Основные усилия стоит направить на:
1. Рефакторинг структуры папок (убрать `screens`, всё в `features`).
2. Улучшение обработки ошибок (не возвращать `false`, а прокидывать исключения).
3. Упрощение инициализации в `main.dart`.
